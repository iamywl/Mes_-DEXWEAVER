## 1. 목적 및 범위 (Purpose and Scope)

### 1.1 목적 (Purpose)

본 문서는 DEXWEAVER MES v4.3 시스템에 대한 **검증 및 확인(Verification and Validation, V&V)** 계획을 IEEE 1012-2016 표준에 따라 수립한다. V&V 활동을 통해 다음 목표를 달성한다.

- 소프트웨어가 요구사항 명세에 부합하게 구현되었는지 **검증(Verification)** 한다.
- 소프트웨어가 사용자 요구 및 운영 환경에 적합한지 **확인(Validation)** 한다.
- 소프트웨어 무결성 수준(Software Integrity Level)에 상응하는 V&V 엄격도를 적용한다.
- 전체 소프트웨어 수명주기에 걸쳐 체계적이고 추적 가능한 V&V 활동을 수행한다.

### 1.2 범위 (Scope)

본 V&V 계획의 대상 범위는 다음과 같다.

| 구분 | 대상 |
|------|------|
| 백엔드 서버 | FastAPI 기반 REST API (40+ 엔드포인트, 24개 모듈, ~4,726 LoC) |
| 프론트엔드 | React 19 + Vite + Tailwind CSS 4 SPA (~2,850 LoC) |
| 데이터베이스 | PostgreSQL 15 (12개 테이블, init.sql 스키마) |
| AI 엔진 | 수요예측, 스케줄최적화, 불량예측, 설비고장예측 (4종) |
| 컨테이너 인프라 | Docker, docker-compose, Kubernetes 매니페스트 |
| 요구사항 | 기능 38건(FN), 비기능 12건(NF), 보안 12건(SEC) = 총 62건 |

### 1.3 V&V 계획 적용 제외 사항

- 하드웨어 설비(PLC, 센서 등)의 물리적 검증은 본 계획에서 제외한다.
- 외부 COTS(Commercial Off-The-Shelf) 소프트웨어(PostgreSQL, Docker, Kubernetes)의 내부 검증은 범위에 포함하지 않는다.
- 네트워크 인프라 및 물리 보안은 별도 계획에서 다룬다.

## 5. 소프트웨어 무결성 수준 분류 (SIL Classification)

### 5.1 DEXWEAVER MES SIL 등급: **SIL 2 (Major — 가역적 영향)**

### 5.2 SIL 분류 근거 (Justification)

DEXWEAVER MES v4.3은 스마트 팩토리 제조실행시스템으로서 다음 분석에 기반하여 **SIL 2**로 분류한다.

#### 5.2.1 결함 영향 분석 (Failure Impact Analysis)

| 영향 범주 | 결함 시 예상 결과 | 심각도 |
|-----------|----------------|--------|
| 인명 안전 | MES는 설비를 직접 제어하지 않으며, PLC/인터록이 별도 작동. 인명 피해 가능성 극히 낮음 | 낮음 |
| 경제적 영향 | 생산 계획 오류, 재고 불일치, 품질 관리 실패로 인한 생산 중단 및 불량 발생. 복구 가능 | **중간~높음** |
| 운영 영향 | LOT 추적 실패 시 리콜 대응 지연, 작업지시 오류 시 생산라인 일시 중단 | 중간 |
| 데이터 무결성 | BOM, 재고, 품질 데이터 오류 시 후속 공정 연쇄 영향. DB 백업으로 복구 가능 | 중간 |
| AI 엔진 영향 | 예측 부정확 시 비최적 의사결정. 인간 판단으로 보완 가능 | 낮음~중간 |

#### 5.2.2 SIL 2 선정 논리

1. **인명 안전 비관여**: MES는 정보 시스템이며 설비를 직접 제어하지 않으므로 SIL 4(Critical)에 해당하지 않는다.
2. **경제적 피해 발생 가능**: 생산 중단, 불량 발생, 재고 불일치는 상당한 경제적 손실을 유발한다.
3. **복구 가능**: 모든 데이터는 PostgreSQL에 저장되며, 트랜잭션 롤백 및 DB 백업을 통해 복구 가능하다.
4. **대체 수단 존재**: MES 장애 시 수동 생산 관리로 일시적 전환이 가능하다.
5. **AI 보조적 역할**: 4종 AI 엔진은 의사결정 보조 도구이며, 최종 결정은 운영자가 수행한다.

따라서 "상당한 경제적 영향이 있으나, 복구 가능한 수준"인 **SIL 2**가 적정하다.

**코드 리뷰 중점 항목 (SIL 2 기준)**:
- SQL Injection 방어: 전 모듈 `cursor.execute(query, params)` 패턴 확인
- 비밀번호 처리: bcrypt 해싱 적용 (`$2b$` 접두어)
- JWT 구현: exp 만료, role 필드, 시크릿 환경변수 관리
- 전역 예외 처리: 스택트레이스 미노출, 한국어 에러 메시지
- DB 커넥션 관리: `db_connection()` 컨텍스트 매니저 활용

| 운영 V&V 보고서 | 월 1회 | 운영 중 V&V 활동 결과, 인시던트 분석 | V&V 관리자, 운영팀 |

### 8.2 보고서 필수 포함 항목

각 V&V 보고서는 다음 항목을 반드시 포함하여야 한다.

1. **보고서 식별**: 문서 ID, 버전, 작성일, 작성자
2. **대상 식별**: 대상 소프트웨어 버전, 형상 기준선
3. **V&V 활동 요약**: 수행한 V&V 태스크 목록, 적용 기법
4. **결과 요약**: 합격/불합격 판정, 발견 이상 건수 및 심각도 분포
5. **이상 목록**: 각 이상의 ID, 설명, 심각도, 상태, 시정 조치
6. **추적성 현황**: RTM 기반 추적 완전성 수치
7. **결론 및 권고**: V&V 관점의 종합 판정, 잔여 위험, 권고 사항

## 9. V&V 행정 절차 (V&V Administrative Procedures)

### 9.1 이상 해결 절차 (Anomaly Resolution)

V&V 활동 중 발견된 이상(Anomaly)은 다음 절차에 따라 관리한다.

#### 9.1.1 이상 심각도 분류

| 심각도 | 정의 | 조치 기한 | 예시 (MES 맥락) |
|--------|------|----------|----------------|
| Critical | 시스템 운영 불가, 데이터 손실 위험 | 24시간 이내 | DB 트랜잭션 오류, 인증 우회 |
| Major | 핵심 기능 불가, 우회 방법 없음 | 72시간 이내 | LOT 추적 실패, BOM 정전개 오류 |
| Minor | 기능 부분 제한, 우회 가능 | 다음 릴리스 | HTTP 상태코드 비표준 사용 (T-008) |
| Info | 개선 권고, 기능 영향 없음 | 권고 사항 | FK 제약 오류 메시지 형식 |

#### 9.1.2 이상 처리 흐름

```
발견 → 보고(이상 보고서) → 분류(심각도) → 분석(원인) → 시정 조치 → 검증(재테스트) → 종료
```

1. **발견(Detection)**: V&V 태스크 수행 중 이상 식별
2. **보고(Reporting)**: V&V 이상 보고서 작성, 고유 ID 부여
3. **분류(Classification)**: 심각도 및 우선순위 결정
4. **분석(Analysis)**: 근본 원인 분석 (Root Cause Analysis)
5. **시정 조치(Corrective Action)**: 개발팀의 수정 이행
6. **검증(Verification)**: V&V 팀의 시정 조치 확인 (재테스트)
7. **종료(Closure)**: V&V 관리자 승인 후 이상 종료

### 9.2 예외 처리 절차 (Exception Handling)

V&V 계획에서 정의된 활동을 수행할 수 없거나, SIL 2 기준의 V&V 엄격도를 완화하여야 하는 경우 다음 절차를 따른다.

1. **예외 요청**: 예외 사유를 명시하여 V&V 관리자에게 서면 요청
2. **영향 분석**: 예외가 소프트웨어 무결성에 미치는 영향 분석
3. **대안 수립**: 대체 V&V 활동 또는 보완 조치 제안
4. **승인**: V&V 관리자 및 프로젝트 관리자의 공동 승인
5. **기록**: 예외 사항, 사유, 대안, 승인 내역을 V&V 보고서에 기록

**예외 허용 기준**: SIL 2 소프트웨어에서 V&V 활동의 예외는 소프트웨어 무결성 수준을 저하시키지 않는 범위에서만 허용된다.

### 9.3 형상 관리 인터페이스 (Configuration Management Interface)

V&V 활동은 프로젝트 형상 관리(CM) 체계와 다음과 같이 연동한다.

| CM 활동 | V&V 연동 사항 |
|---------|-------------|
| 형상 식별 | V&V 대상 형상 항목(소스코드, 문서, 테스트 케이스) 식별 및 등록 |
| 형상 기준선 수립 | 각 단계 전환 시 V&V 완료를 기준선 수립 전제 조건으로 설정 |
| 변경 통제 | 변경 요청(CR) 발생 시 V&V 영향 분석 필수 수행 |
| 형상 감사 | V&V 산출물의 형상 감사 참여, 기준선 무결성 확인 |
| 상태 보고 | V&V 이상 현황을 형상 상태 보고에 포함 |

**Git 기반 형상 관리 현황**:
- 버전 관리: Git (Conventional Commits 준수)
- 현재 커밋: 20+ 건
- 형상 항목: 소스코드 24개 모듈, 프론트엔드 SPA, DB 스키마, Docker 설정, K8s 매니페스트
