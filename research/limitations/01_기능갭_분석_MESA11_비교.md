# DEXWEAVER MES 기능 갭(Gap) 분석 보고서

> **작성일**: 2026-03-03
> **대상 시스템**: DEXWEAVER MES v4.0
> **분석 기준**: MESA-11 모델, ISA-95 표준, 글로벌 경쟁사(Siemens Opcenter, Rockwell Plex, SAP DMC, DELMIA Apriso, Honeywell MXP, Critical Manufacturing)
> **분석 범위**: Backend(app.py + 18개 api_modules), Frontend(App.jsx 2,851줄), DB(21개 테이블), Infra(K8s/Cilium)

## 1. 개요

### 1.1 분석 목적

본 보고서는 현재 개발 중인 DEXWEAVER MES의 기능 완성도를 MESA(Manufacturing Enterprise Solutions Association)가 정의한 MES 11대 핵심 기능 모델에 기반하여 체계적으로 평가하고, 업계 표준 및 글로벌 경쟁사 대비 기능적 갭(Gap)을 식별하여 구체적인 개선 로드맵을 제시하는 것을 목적으로 한다.

### 1.2 비교 기준 프레임워크

| 기준 | 설명 |
|------|------|
| **MESA-11** | MES의 11대 핵심 기능 정의 (2024 업데이트 반영) |
| **ISA-95 (IEC 62264)** | 제조업 정보 시스템 통합 표준 - Level 0~4 계층 모델 |
| **ISA-88 (IEC 61512)** | 배치 프로세스 제어 표준 - 레시피 관리 |
| **FDA 21 CFR Part 11** | 전자기록/전자서명 규제 (제약/의료기기) |
| **GAMP 5** | GxP 컴퓨터 시스템 밸리데이션 가이드라인 |

### 1.3 평가 등급 체계

| 등급 | 의미 | 점수 |
|------|------|------|
| **A (Excellent)** | 업계 표준 이상, 경쟁사 동등 수준 | 90~100 |
| **B (Good)** | 핵심 기능 구현, 일부 고급 기능 부재 | 70~89 |
| **C (Fair)** | 기본 기능만 구현, 상당한 보완 필요 | 50~69 |
| **D (Poor)** | 최소 기능만 존재, 근본적 개선 필요 | 30~49 |
| **F (Missing)** | 미구현 또는 극히 초기 단계 | 0~29 |

### 3.1 생산 자원 배분 및 상태 (Resource Allocation & Status)

#### 3.1.1 현재 구현 상태

**구현된 부분:**
- `mes_equipment.py`: 설비(Equipment) CRUD 및 상태 관리 (RUNNING/STOP/DOWN)
- `equip_status_log` 테이블: 설비 상태 변경 이력 관리
- `mes_equipment.get_equipment_status()`: 일별 가동률(uptime_today) 자동 계산
- 설비별 시간당 생산능력(capacity_per_hour) 관리
- 작업지시(work_orders)와 설비 연결을 통한 현재 작업 표시

**코드 레벨 문제점:**

1. **설비 자원만 관리, 나머지 자원 유형 미지원**: `equipments` 테이블은 물리적 설비만 관리한다. 작업자(Operator), 금형(Mold/Die), 치공구(Jig/Fixture), 측정기, 원부자재 등의 자원은 관리 대상에 포함되지 않는다.

2. **자원 자동 배정 로직 부재**: `mes_work.py`의 `create_work_order()`에서 `equip_code`를 수동으로 입력받는다. 설비 가용성, 능력, 적합성을 고려한 자동 배정 알고리즘이 없다.

```python
# mes_work.py:46 - 설비 코드를 단순히 사용자 입력으로 받음
(wo_id, plan_id, plan[0], work_date,
 data.get("equip_code"), data.get("plan_qty", plan[1])),
```

3. **자원 가용성 실시간 조회 불가**: 특정 시점에 어떤 설비가 사용 가능한지를 쿼리하는 API가 없다. `get_equipments()`는 현재 상태만 반환할 뿐, 시간 슬롯별 가용성(Availability Calendar)을 제공하지 않는다.

4. **자원 능력(Capability) 정보 미관리**: 어떤 설비가 어떤 품목/공정을 처리할 수 있는지에 대한 매핑이 `processes.equip_code` 단일 컬럼으로만 관리된다. 다대다(M:N) 관계를 지원하지 않아, 하나의 설비가 여러 공정에 사용 가능한 상황을 표현할 수 없다.

### 3.2 작업 스케줄링 (Operations/Detail Scheduling)

#### 3.2.1 현재 구현 상태

**구현된 부분:**
- `mes_plan.py`: OR-Tools CP-SAT 제약 프로그래밍 기반 스케줄러 (`_ortools_schedule()`)
- 다기계 할당 + 비중복 제약 + 우선순위 가중치 지원
- Greedy 휴리스틱 폴백 (`_greedy_schedule()`)
- Prophet 기반 수요예측 (`mes_ai_prediction.py`)
- 생산계획 CRUD + 날짜/상태 필터링

**코드 레벨 문제점:**

1. **단일 목적 최적화**: 현재 `_ortools_schedule()`은 `makespan * 10 + sum(priority_cost)`만 최적화한다. 실제 제조 환경에서는 납기 준수율, 셋업 시간 최소화, 재고 비용, 설비 가동률 균형 등 다목적(Multi-Objective) 최적화가 필요하다.

```python
# mes_plan.py:305 - 단일 목적함수만 사용
model.minimize(makespan * 10 + sum(priority_cost))
```

2. **셋업 타임(Setup/Changeover) 미고려**: 동일 설비에서 품목 변경 시 발생하는 셋업 시간이 모델에 반영되지 않는다. 실제 제조에서 셋업 타임은 총 리드타임의 20~50%를 차지한다.

3. **제약 조건 부족**: 현재 구현에는 다음 제약이 없다:
   - 자재 가용성 (소요 원자재가 재고에 있는지)
   - 설비 정비 일정 (정비 예정 시간 회피)
   - 작업자 가용성 (교대 근무 패턴)
   - 공정 간 선후행 관계 (Routing 순서)
   - 최소/최대 배치 크기

4. **실시간 리스케줄링 불가**: 일회성(Batch) 스케줄링만 지원한다. 설비 고장, 긴급 주문, 자재 지연 등 예외 상황에서 실시간으로 스케줄을 조정하는 Event-Driven Rescheduling이 없다.

5. **유한 능력 스케줄링(FCS) 미완성**: 설비의 `capacity_per_hour`만 고려하고, 교대 근무 패턴, 계획 정비, 설비 고장률 등을 고려한 실제 가용 능력(Available Capacity)을 계산하지 않는다.

```python
# mes_plan.py:218-219 - capacity_per_hour만으로 단순 계산
cap = e[2] if e[2] > 0 else 1
dur = max(1, int(qty / (cap / 60)))
```

**단기 (1~3개월):**

**중기 (3~6개월):**
- 다목적 최적화 구현: Pareto Front 기반 납기/가동률/셋업 Trade-off 분석
- What-If 시나리오 엔진: 복수 스케줄 생성 + 비교 API
- 이벤트 드리븐 리스케줄링: WebSocket 기반 설비 고장/긴급주문 감지 -> 자동 리스케줄
- 간트 차트 고도화: 드래그&드롭 수동 조정 + 실시간 KPI 반영

**장기 (6~12개월):**

### 3.7 품질 관리 (Quality Management)

#### 3.7.1 현재 구현 상태

**구현된 부분:**
- `mes_quality.py`: 검사 기준 등록(NUMERIC/TEXT/VISUAL 타입), 검사 실행 + 자동 PASS/FAIL 판정
- `mes_reports.py`: Cpk 계산(실측치 기반), X-bar 관리도 데이터, 불량률 추이
- `mes_defect_predict.py`: XGBoost+SHAP 기반 불량 확률 예측
- `defect_codes` 테이블: 불량 유형 코드 관리
- 불량 현황/추이 API

**코드 레벨 문제점:**

1. **SPC(Statistical Process Control) 미구현**: `mes_reports.py`에서 Cpk와 X-bar 관리도 데이터를 계산하지만, 실시간 SPC 모니터링 시스템이 아니다. 다음이 부재하다:
   - 관리도 종류: X-bar/R, X-bar/S, p, np, c, u 차트
   - Western Electric Rules / Nelson Rules 기반 이상 패턴 감지
   - 관리 한계 자동 재계산
   - 공정능력 지수(Cp, Cpk, Pp, Ppk) 대시보드

2. **CAPA(Corrective and Preventive Action) 프로세스 없음**: 불량 발생 시 원인 분석 -> 시정조치 -> 재발방지 -> 효과확인의 체계적 워크플로우가 없다.

3. **FMEA(Failure Mode and Effects Analysis) 미지원**: 잠재적 고장 모드와 영향을 사전에 분석하는 도구가 없다.

4. **MSA(Measurement System Analysis) 없음**: 측정 시스템의 정확도/정밀도를 평가하는 Gage R&R 분석이 없다.

5. **Cpk 폴백 로직의 비통계적 계산**:

```python
# mes_reports.py:235 - 통계적으로 의미 없는 폴백 Cpk
if cpk is None:
    cpk = round(max(0, (1 - defect_rate * 10) * 1.33), 2)
```

이 폴백 공식은 실제 공정능력과 무관한 추정값이다. 데이터가 없을 경우 Cpk를 제공하지 않는 것이 올바르다.

6. **검사 유형 제한**: 현재 PROCESS(공정 검사)만 지원한다. IQC(수입검사), FQC(최종검사), OQC(출하검사) 등 검사 유형별 분리 관리가 부족하다.

#### 3.7.2 업계 표준/경쟁사 수준

| 경쟁사 | 품질 관리 수준 |
|--------|-------------|

**코드 레벨 문제점:**

1. **OEE(Overall Equipment Effectiveness) 미구현**: 제조업의 가장 중요한 KPI인 OEE = 가동률(A) x 성능(P) x 양품률(Q)이 자동 계산되지 않는다. 현재 `uptime_today`(가동률)만 개별적으로 계산되고, 성능(이론 대비 실제 사이클 타임)과 양품률은 OEE와 통합되지 않는다.

2. **KPI 대시보드 부족**: 총 11개의 KPI만 산출(`mes_reports.ai_insights()`). 경쟁사(Rockwell Plex)는 2,600+ KPI를 제공한다. 다음 핵심 KPI가 부재하다:
   - OEE (설비종합효율)
   - MTBF (평균고장간격)
   - MTTR (평균수리시간)
   - Yield Rate (수율)
   - Throughput (처리량)
   - WIP (Work-in-Progress) 수준
   - Cycle Time vs Takt Time
   - 납기 준수율 (On-Time Delivery)
   - 자재 회전율 (Inventory Turnover)

3. **실시간 대시보드 부재**: 모든 데이터가 API 호출 시점의 스냅샷이다. WebSocket 기반 실시간 업데이트가 없어, 대시보드를 새로고침해야 최신 데이터를 볼 수 있다.

4. **BI(Business Intelligence) 연동 없음**: Power BI, Tableau, Grafana 등 외부 BI 도구와의 연동 인터페이스가 없다.

5. **리포트 내보내기 없음**: PDF, Excel, CSV 등으로 리포트를 내보내는 기능이 없다.

#### 3.11.2 업계 표준/경쟁사 수준

| 경쟁사 | 성과 분석 수준 |
|--------|-------------|
| **Rockwell Plex** | 2,600+ 사전 정의 KPI. 실시간 대시보드. 커스텀 리포트 빌더. |
| **Siemens Opcenter** | 실시간 OEE. 트렌드 분석. Drilldown 분석. MindSphere Analytics 연동. |
| **SAP DMC** | SAP Analytics Cloud 연동. AI 기반 이상 탐지. 예측 분석. |

#### 3.11.3 갭 분석

| 갭 항목 | 심각도 | 설명 |
|---------|--------|------|
| OEE 자동 계산 없음 | CRITICAL | 제조업 핵심 KPI 부재 |

#### 3.11.4 개선 제안

**단기 (1~3개월):**
- **OEE 자동 계산 엔진 구현**:
  - 가동률(A) = 실가동시간 / 계획가동시간 (equip_status_log 활용)
  - 성능(P) = 이론 사이클타임 x 실생산량 / 실가동시간 (routings.cycle_time 활용)
  - 양품률(Q) = 양품수 / 총생산수 (work_results 활용)
  - `GET /api/kpi/oee?equip_code=EQP-001&date=2026-03-03`
- MTBF/MTTR 자동 계산 (equip_status_log 기반)
- 납기 준수율 계산 (production_plans.due_date vs 실제 완료일)
- 리포트 PDF/Excel Export API (WeasyPrint/openpyxl)

**중기 (3~6개월):**
- WebSocket 기반 실시간 대시보드
- KPI 라이브러리 확장 (50+ 표준 KPI)
- Grafana 연동 (PostgreSQL 직접 연결 또는 Prometheus Exporter)
- 커스텀 대시보드 빌더 (드래그&드롭 위젯)

**장기 (6~12개월):**
- BI 플랫폼 연동 (Power BI Embedded, Superset)
- AI 기반 이상 탐지 고도화 (Prophet 시계열 분해)
- 벤치마킹 포털 (다공장/다라인 성과 비교)
- 경영진용 Executive Dashboard

### 6.3 DEXWEAVER의 차별화 요소 (현재 강점)

현재 시스템이 경쟁사 대비 차별점을 가진 영역:

1. **AI 통합 수준**: 4가지 AI 모델(Prophet, XGBoost/SHAP, IsolationForest, OR-Tools)이 백엔드에 네이티브 통합. 경쟁사는 대부분 외부 AI 플랫폼 연동.

2. **클라우드 네이티브 인프라**: Kubernetes + Cilium eBPF 기반. 네트워크 수준 가시성(Hubble). 경쟁사 중 Critical Manufacturing만 유사 수준.

3. **경량 스택**: Python FastAPI + React 단일 스택으로 빠른 개발/배포. 대형 경쟁사 대비 도입 비용/복잡성 현저히 낮음.

4. **오픈 아키텍처**: 특정 벤더 Lock-in 없음. 오픈소스 기반으로 커스터마이징 자유도 높음.

### 6.4 핵심 전략 권고

1. **"AI-First MES"로 포지셔닝**: AI가 이미 핵심에 통합된 유일한 경량 MES로 차별화. Phase 1에서 SPC+AI, CBM+AI 등 AI 적용 범위 확대.

2. **수직 산업 특화**: 모든 산업을 커버하기보다 1~2개 타겟 산업(예: 전자/반도체, 제약/바이오)에 특화. 해당 산업의 규제/표준 우선 준수.

3. **개발자 경험(DX) 우선**: API-First 접근 + 풍부한 문서 + 쉬운 커스터마이징으로 중소 제조기업 시장 공략.

4. **점진적 확장**: Phase 1 -> 2 -> 3 순서를 엄격히 준수. 기본기(SPC, CAPA, OEE, CMMS)를 먼저 완성한 후 고급 기능 추가.
