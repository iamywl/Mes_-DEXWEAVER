## Executive Summary

DEXWEAVER MES는 FastAPI + React 기반의 제조 실행 시스템으로, 21개 테이블 기반의 핵심 MES 기능(FN-001~FN-038)을 구현하고 있다. AI/ML 모듈(Prophet, XGBoost, IsolationForest, OR-Tools)을 포함하며, Kubernetes 환경에 배포된다. 그러나 현재 아키텍처는 **프로토타입 수준의 모놀리식 구조**에 머물러 있어, 엔터프라이즈 MES로의 확장에 구조적 한계가 존재한다.

**핵심 문제**: 프론트엔드 단일 파일 2,851줄, 백엔드 인증 보일러플레이트 반복, DB 마이그레이션 부재, AI 모델 매 요청시 재학습, ConfigMap 기반 배포 등 -- 이 모든 것이 **운영 안정성, 확장성, 유지보수성**에 직접적 위험 요소이다.

**종합 성숙도**: 경쟁사 대비 평균 **1.8 / 5.0점** (Siemens Opcenter, Rockwell Plex, Critical Manufacturing 대비)

모든 엔드포인트가 `app.py` 한 파일에 정의되어 있다. FastAPI의 `APIRouter`를 전혀 활용하지 않는다.

#### 2.3 Pydantic 모델 미사용

요청/응답 모두 `dict`로 처리하며, 입력 검증은 각 모듈에서 수동으로 수행한다.

#### 2.4 에러 응답 비표준

에러 발생 시 HTTP 200과 함께 `{"error": "..."}` dict를 반환한다. REST API 표준을 따르지 않으며, 클라이언트에서 에러 감지가 어렵다.

```python
# 현재: HTTP 200 + error dict (비표준)
if not conn:
    return {"error": "Database connection failed."}
# 클라이언트는 response.status === 200이지만 실제로는 에러
```

#### 2.5 커넥션 관리 반복

모든 모듈에서 동일한 `conn = get_conn() / release_conn(conn)` 패턴이 반복된다.

```python
# 현재: 모든 함수에서 반복되는 커넥션 관리 패턴
async def create_equipment(data: dict) -> dict:
    conn = None
    try:
        conn = get_conn()
        if not conn:
            return {"error": "Database connection failed."}
        cursor = conn.cursor()
        # ... 로직 ...
        conn.commit()
        cursor.close()
        return {"success": True}
    except Exception as e:
        if conn:
            conn.rollback()
        return {"error": str(e)}
    finally:
        if conn:
            release_conn(conn)
```

이 패턴이 28개 모듈 전체에서 반복되고 있다.

```python
security = HTTPBearer()

async def get_current_user(

) -> dict:
    payload = _verify_token(credentials.credentials)
    if not payload:
        raise HTTPException(
            status_code=status.HTTP_401_UNAUTHORIZED,
            detail="유효하지 않은 인증 토큰입니다."

async def require_admin(user: dict = Depends(get_current_user)) -> dict:
```

```python
from fastapi import HTTPException

class MESException(HTTPException):
    def __init__(self, status_code: int, detail: str, error_code: str = None):
        super().__init__(status_code=status_code, detail=detail)
        self.error_code = error_code

class NotFoundError(MESException):
    def __init__(self, resource: str, identifier: str):
        super().__init__(
            status_code=404,
            detail=f"{resource} '{identifier}'을(를) 찾을 수 없습니다.",
            error_code="RESOURCE_NOT_FOUND"
        )

class DatabaseError(MESException):
    def __init__(self):
        super().__init__(
            status_code=503,
```

스키마 변경 시 `init.sql`을 직접 수정하며, 버전 관리가 없다. 운영 환경에서의 스키마 변경이 데이터 유실 위험을 동반한다.

```sql
-- 현재: init.sql에 스키마와 시드 데이터가 혼재
CREATE TABLE IF NOT EXISTS users ( ... );
-- ... 21개 테이블 DDL ...

-- 바로 이어서 시드 데이터

-- ... 수백 줄의 INSERT 문 ...
```

#### 3.2 인덱스 부재

21개 테이블에 PK와 UNIQUE 제약 외에 **명시적 인덱스가 없다**. 빈번한 조회 컬럼에 대한 인덱스 미생성으로 데이터 증가 시 성능 저하가 예상된다.

```sql
-- 현재: 인덱스 없이 빈번하게 조회되는 패턴
-- mes_plan.py에서:
"SELECT ... FROM production_plans p WHERE p.due_date >= %s AND p.status = %s"
-- due_date, status 컬럼에 인덱스 없음

-- mes_work.py에서:
"SELECT ... FROM work_orders WHERE work_date = %s AND equip_code = %s"
-- work_date, equip_code에 복합 인덱스 없음

-- mes_equipment.py에서:
"SELECT ... FROM equip_status_log WHERE equip_code = %s ORDER BY changed_at DESC"
-- (equip_code, changed_at) 복합 인덱스 없음
```

#### 3.3 Raw SQL 중복 및 유지보수 어려움

모든 모듈에서 Raw SQL을 직접 작성하며, 동일 테이블에 대한 유사한 쿼리가 여러 모듈에 분산되어 있다.

```python
# mes_plan.py -- f-string으로 SQL 동적 생성
cursor.execute(
    f"SELECT p.plan_id, i.name, p.plan_qty, p.due_date, "
    f"p.status, p.priority, "
    f"COALESCE(SUM(wr.good_qty), 0) AS done_qty "
    f"FROM production_plans p "
    f"JOIN items i ON p.item_code = i.item_code "
    f"LEFT JOIN work_orders wo ON p.plan_id = wo.plan_id "
    f"LEFT JOIN work_results wr ON wo.wo_id = wr.wo_id "
    f"{where_sql} "                    # ← f-string에 동적 SQL 삽입
    f"GROUP BY p.plan_id, i.name ... "
    f"LIMIT 20 OFFSET %s",
    params + [offset],
)
```

`where_sql`은 `" AND ".join(where)` 형태로 f-string에 삽입되지만, 파라미터화된 쿼리와 혼합 사용하고 있어 가독성이 떨어진다.

#### 3.4 커넥션 풀 규모 부족

```python
# 개선: ml/model_registry.py
import pickle
import hashlib
from datetime import datetime, timedelta
from pathlib import Path

class ModelRegistry:
    def __init__(self, cache_dir: str = "/app/ml_cache"):
        self.cache_dir = Path(cache_dir)
        self.cache_dir.mkdir(parents=True, exist_ok=True)
        self._memory_cache = {}
        self._max_age = timedelta(hours=6)  # 6시간마다 재학습

    def get_model(self, model_key: str) -> tuple:
        """캐시에서 모델 조회, 없거나 만료되면 None 반환."""
        if model_key in self._memory_cache:
            model, timestamp = self._memory_cache[model_key]
            if datetime.now() - timestamp < self._max_age:
                return model

        # 디스크 캐시 확인
        cache_path = self.cache_dir / f"{model_key}.pkl"
        if cache_path.exists():
            with open(cache_path, "rb") as f:
                model, timestamp = pickle.load(f)
            if datetime.now() - timestamp < self._max_age:
                self._memory_cache[model_key] = (model, timestamp)
                return model
        return None

    def save_model(self, model_key: str, model):
        """모델을 메모리 + 디스크에 캐싱."""
        timestamp = datetime.now()
        self._memory_cache[model_key] = (model, timestamp)
        cache_path = self.cache_dir / f"{model_key}.pkl"
        with open(cache_path, "wb") as f:
            pickle.dump((model, timestamp), f)

registry = ModelRegistry()


# 개선된 mes_defect_predict.py
async def predict_defect_probability(process_params: dict) -> dict:
    model_key = "defect_xgboost_v1"

    # 캐시된 모델 사용
    cached_model = registry.get_model(model_key)
    if cached_model:
        model = cached_model
    else:
        # 백그라운드 태스크로 학습
        X_train, y_train = await _load_training_data()
        model = xgb.XGBRegressor(n_estimators=100, ...)

        registry.save_model(model_key, model)

    # 예측만 수행 (빠름)
```

### 한계점

#### 5.1 ConfigMap 기반 배포 (Critical)

Docker 이미지를 빌드하지 않고, 소스코드를 ConfigMap에 저장한 뒤 컨테이너에 마운트한다. 이는 매우 비정상적인 배포 방식이다.

```yaml
# 현재: mes-api.yaml
containers:
  - name: mes-api
    image: python:3.9-slim        # ← 베이스 이미지만 사용
    command: ["/bin/sh", "-c"]
    args:
      - |
        mkdir -p /app/api_modules &&
        cp /mnt/*.py /app/api_modules/ &&       # ← ConfigMap에서 소스 복사
        mv /app/api_modules/app.py /app/app.py &&
        touch /app/api_modules/__init__.py &&
        pip install --cache-dir /pip-cache fastapi uvicorn psycopg2-binary ... &&  # ← 매번 설치
        pip install --cache-dir /pip-cache numpy scikit-learn xgboost shap ortools 2>/dev/null || true &&
        pip install --cache-dir /pip-cache prophet 2>/dev/null || true &&
        python /app/app.py
```

**문제점**:
1. **시작 시간**: 매 Pod 시작마다 pip install 실행 (수분 소요)
2. **ConfigMap 크기 제한**: 1MB, 코드 크기가 커지면 배포 불가
3. **재현 불가능**: 빌드 산출물이 없어 특정 버전 재현 불가
4. **보안 위험**: pip 설치 시 네트워크 접근 필요, supply chain 공격에 취약
5. **롤백 불가**: 이미지 태그 기반 롤백 불가능

#### 5.2 단일 복제본 (No HA)

모든 컴포넌트가 replicas: 1로 설정되어 있어 단일 장애 지점(SPOF)이다.

```yaml
# 현재: 모든 Deployment에서 replicas: 1
spec:
  replicas: 1   # ← API, Frontend, PostgreSQL 모두 1개
```

### 한계점

#### 8.1 동기 DB 드라이버 + 비동기 프레임워크 불일치

FastAPI는 비동기 프레임워크이지만, `psycopg2`는 동기 드라이버이다. `async def` 함수 내에서 동기 DB 호출이 발생하면 이벤트 루프가 블로킹된다.

```python
# 현재: async 함수에서 동기 DB 호출
async def get_items(keyword, category, page, size):
    conn = get_conn()       # ← 동기 호출 (이벤트 루프 블로킹)
    cursor = conn.cursor()  # ← 동기 호출
    cursor.execute(...)     # ← 동기 호출 (쿼리 완료까지 블로킹)
```

#### 8.2 캐싱 레이어 부재

마스터 데이터(품목, 공정, 설비 등)에 대한 캐싱이 없어, 모든 요청이 DB를 직접 조회한다.

#### 8.3 API 응답 압축 없음

대량 데이터 조회 시 응답 압축(gzip)이 설정되어 있지 않아 네트워크 대역폭을 비효율적으로 사용한다.

#### 8.4 프론트엔드 번들 최적화 없음

단일 App.jsx가 전체를 포함하므로 코드 스플리팅이 불가능하며, 초기 로드 시 ~179KB의 JSX 전체를 다운로드해야 한다.

#### 8.5 페이지네이션 불일치

일부 엔드포인트만 페이지네이션을 지원하며, 나머지는 전체 데이터를 반환한다.

### 결론

DEXWEAVER MES는 핵심 MES 기능(FN-001~FN-038)을 구현한 **기능적으로 완성된 프로토타입**이지만, 아키텍처적으로는 **엔터프라이즈 운영 수준에 미달**한다. 종합 성숙도 1.6/5.0으로, 경쟁사 평균(4.2/5.0)과 큰 격차가 있다.

가장 시급한 조치는 **Phase 1의 ConfigMap 배포 제거, JWT Secret 고정화, K8s 프로브/리소스 설정**이며, 이는 1~2주 내에 완료 가능하다. Phase 2의 프론트엔드 분리와 백엔드 리팩토링은 코드 품질과 개발 생산성에 직접적 영향을 미치므로 병행 추진이 권장된다.

AI/ML 모듈은 현재 **graceful fallback 패턴**이 잘 구현되어 있어 기능적 안정성은 확보되어 있으나, 매 요청시 학습 방식은 반드시 캐싱 기반으로 전환해야 한다.

장기적으로 경쟁사 수준(4.0+)에 도달하기 위해서는 **이벤트 드리븐 아키텍처, IoT 연동, MLOps 파이프라인**의 도입이 필수적이며, 이는 Phase 3에서 전략적으로 추진해야 한다.
